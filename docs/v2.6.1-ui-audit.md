## 2. 问题清单

### 2.1 TimelineView 问题

#### P1 · MultiSelectDropdown 位置偏移 / 截断

- **表现**：筛选栏的下拉框使用绝对定位 `top-full left-0 mt-1`，始终向左对齐触发按钮。当触发按钮靠近视口右边缘，或下拉内容较长（角色/地点列表）时，下拉面板右侧容易超出视口被截断；窄屏下尤其明显。
- **截图证据**：`03-timeline-filter-status-open.png`（下拉左对齐，右侧安全距离不足）
- **根因代码**：`src/components/views/TimelineView.tsx` 第 2395–2401 行，自定义 `MultiSelectDropdown` 未做视口碰撞检测（flip/shift），也未使用 Portal 脱离可能带 `overflow: hidden` 的祖先容器。
- **影响文件**：
  - `src/components/views/TimelineView.tsx`
  - 可替换为 Radix Popover / DropdownMenu（仓库已依赖 `@radix-ui/react-popover`）

#### P1 · “Today” 标记标签在画布左边缘被裁剪 / 与轨道色标重叠

- **表现**：`DateRuler` 中的 Today 标签使用 `left: todayX` + `-translate-x-1/2` 居中。当 `todayX` 靠近画布左边缘（事件发生在时间范围早期，或用户横向滚动到 Today 线附近），标签左半部分被裁剪，视觉上与左侧轨道色标/轨道标题区发生重叠。
- **截图证据**：`01b-timeline-today-marker.png`（Today 标签紧贴画布左缘，左侧被裁切）
- **根因代码**：`src/components/views/TimelineView.tsx` 第 953–963 行，Today 标签未设置最小左内边距，也未做边界保护（如 `max(left, 20px)` 或改为左对齐）。
- **影响文件**：`src/components/views/TimelineView.tsx`

#### P1 · 事件卡片内部拥挤：`+` 按钮 / “相对”标签 / 标题

- **表现**：
  - 事件卡片固定宽度 `180px`，标题行同时容纳状态点、标题（`truncate`）、“相对”标签、悬停才出现的连线手柄，空间紧张。
  - 长标题或带“相对”标签时，标题可显示字符数进一步减少，几乎只剩标签。
  - 轨道左侧的 `+` 添加按钮（`width: 72px`）与第一个事件卡片默认位置过近，在事件密集或第一个事件为相对事件（靠左排列）时容易重叠或被遮挡。
- **截图证据**：`01-timeline-overview.png`、`06-timeline-narrow-today.png`
- **根因代码**：
  - 卡片宽度与内边距：`src/components/views/TimelineView.tsx` 第 84、1528–1531 行（`EVENT_MIN_WIDTH = 180`、`px-3`、标题行 `gap-1.5`）
  - “相对”标签：`src/components/views/TimelineView.tsx` 第 1550–1554 行
  - `+` 按钮：`src/components/views/TimelineView.tsx` 第 1363–1377 行（固定 `left: 8`，与事件卡片同层级，无避让逻辑）
- **影响文件**：`src/components/views/TimelineView.tsx`

### 2.2 SettingsView 问题

#### P2 · 外观主题卡片布局不对称

- **表现**：主题卡片使用 `grid grid-cols-3 gap-3`，但 `THEMES` 数组只有 4 个主题。第二行仅剩下 1 个“方块”主题卡片，左对齐导致整体视觉重心偏移、布局不饱满。
- **截图证据**：`04-settings-appearance.png`
- **根因代码**：`src/components/views/SettingsView.tsx` 第 100–105 行（4 个主题）、第 280 行（`grid-cols-3`）
- **影响文件**：`src/components/views/SettingsView.tsx`

#### P1 · 字体主题“像素”实际首选项为 Smiley Sans（得意黑）

- **表现**：外观页“界面字体主题”中“像素”选项的字样与实际字体栈不符。点击后应用的 `FONT_THEME_STACKS.pixel.ui` 首字体为 `"Smiley Sans"`（得意黑，一款几何斜黑体，并非像素字体），而真正的像素字体 `"Fusion Pixel 10px"` 只是 fallback。
- **进一步发现**：`src/styles/tailwind.css` 第 47 行将 `--font-pixel` 定义为 `"Smiley Sans", "Fusion Pixel 10px", ...`；但 `src/styles/themes.css` 第 15、96 行又将 `--font-pixel` 重新定义为 `"Fusion Pixel 10px", "Zpix", ...`。同一 CSS 变量在不同文件语义不一致，导致“像素”主题行为取决于具体使用哪个变量。
- **截图证据**：`04-settings-appearance.png`（“像素”按钮预览看起来是 Smiley Sans）
- **根因代码**：
  - 字体主题栈：`src/components/views/SettingsView.tsx` 第 109–128 行
  - CSS 变量冲突：`src/styles/tailwind.css` 第 47 行、`src/styles/themes.css` 第 15、96 行
  - 标签文案：`src/i18n/locales/zh-CN.json` 第 539 行 `"fontThemePixel": "像素"`
- **影响文件**：`src/components/views/SettingsView.tsx`、`src/styles/tailwind.css`、`src/styles/themes.css`、`src/i18n/locales/zh-CN.json`、`src/i18n/locales/en.json`

#### P1 · AI 设置缺少连接状态与测试按钮

- **表现**：AI 标签页只有启用开关、提供商网格、Base URL、模型下拉、API Key、RAG 开关、系统提示词。用户填写 Base URL + Key + 模型后，无法直观知道当前配置是否可连通；只有在 AI 助手面板真正发送请求失败时才收到错误提示。
- **截图证据**：`05-settings-ai.png`
- **根因代码**：`src/components/views/SettingsView.tsx` 第 568–726 行 AI 标签页，仅有 `modelsQuery` 的刷新/错误状态，缺少显式连接健康检查与测试按钮。
- **影响文件**：
  - `src/components/views/SettingsView.tsx`
  - 可能需要后端/IPC：`src-tauri/src/commands/ai.rs`、`src-tauri/src/services/ai.rs` 增加 `test_ai_connection` 命令
  - 前端 API：`src/features/ai/api.ts`、`src/features/ai/hooks.ts`

## 3. 社区 / 竞品调研总结

### 3.1 时间轴布局与下拉框

- **Notion Timeline**：将事件卡片作为独立块分布在轨道上，卡片宽度随内容自适应，并通过悬停展开详情；筛选器固定在视图顶部，下拉菜单使用原生/Portal 渲染，避免被侧边栏或滚动容器裁剪。
- **Aeon Timeline**：采用左右分栏（轨道列表 + 画布），事件卡片最小宽度更大，标签与标题分层显示；Today / 当前时间标记通常放在标尺顶部，标签做边界保护，不会越过轨道列表。
- **设计惯例**：
  1. 浮动下拉应使用 Portal 并做视口碰撞检测（flip/shift），避免右侧/底部截断。
  2. 时间轴标记标签在靠近边界时应切换为左对齐或添加最小间距，防止裁切。
  3. 事件卡片内部信息层级：标题优先，状态/标签以颜色或微型徽章形式呈现，避免水平堆叠过多元素。

### 3.2 AI 连接状态

- **Claude / Kimi / 豆包 / 通义千问等网页/客户端**：在设置或接入页面普遍提供“连接测试”“检查可用性”按钮，并以绿/黄/红点或文字显示“已连接 / 连接失败 / 未配置”。
- **LobeChat / NextChat / 其他 OpenAI 兼容客户端**：通常会在模型列表旁展示“测试连接”或“验证”按钮，失败时直接给出错误码与重试建议。
- **设计惯例**：API 配置界面应提供即时反馈，降低用户在聊天面板中首次失败才发现配置错误的成本。

### 3.3 字体选择器

- **得意黑（Smiley Sans）**：在中文设计社区中被明确归类为“几何人文黑体”，不是像素字体；其特点是窄、斜、大弧线。
- **Fusion Pixel / Zpix**：才是严格意义上的像素/点阵字体。
- **设计惯例**：字体选项名称应准确反映首选项字体，避免“像素”标签对应非像素字体造成用户困惑；若确实想提供“得意黑”风格，应单独命名为“得意黑”或“几何黑”。
