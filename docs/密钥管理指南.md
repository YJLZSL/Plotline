# 🔐 密钥与签名管理指南

> **必读**：本文档记录 Plotline 自动更新签名密钥的完整管理信息，确保任何接手项目的 AI 或开发者都能快速找到密钥并配置自动更新。

## 密钥位置

| 文件 | 路径 | 用途 | 是否提交到 Git |
|---|---|---|---|
| **私钥** | `keys/plotline.key` | 对安装包进行 Ed25519 签名 | ❌ 否（在 `.gitignore` 中） |
| **公钥** | `keys/plotline.key.pub` | 验证签名，嵌入客户端 | ❌ 否（在 `.gitignore` 中） |
| **公钥（文本）** | `keys/plotline.pub.txt` | 方便复制粘贴的纯文本格式 | ❌ 否（在 `.gitignore` 中） |

> ⚠️ **安全警告**：私钥（`plotline.key`）**绝不可**提交到 GitHub 或分享给任何人。泄露私钥意味着攻击者可以伪造更新包。

## 自动更新配置检查清单

发布新版本时，Release 页面必须包含以下文件：

| 文件 | 来源 | 说明 |
|---|---|---|
| `Plotline_X.Y.Z_x64-setup.exe` | CI 构建 | NSIS 安装程序 |
| `Plotline_X.Y.Z_x64-setup.exe.sig` | CI 签名或手动签名 | Ed25519 签名文件 |
| `Plotline_X.Y.Z_x64_en-US.msi` | CI 构建 | MSI 安装程序 |
| `latest.json` | CI 生成或手动创建 | 自动更新清单（含签名和 URL） |

## 方案 A：配置 GitHub Secret（推荐，一劳永逸）

### 步骤

1. **复制私钥内容**（从 `keys/plotline.key`）：
   ```bash
   cat keys/plotline.key
   # 或直接复制文件内容
   ```

2. **在 GitHub 仓库中配置 Secret**：
   - 打开仓库 Settings → Secrets and variables → Actions
   - 点击 "New repository secret"
   - Name: `TAURI_SIGNING_PRIVATE_KEY`
   - Value: 粘贴 `keys/plotline.key` 的完整内容
   - 点击 "Add secret"

3. **（可选）配置密码**：如果私钥生成时设置了密码
   - Name: `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`
   - Value: 你的密码

4. **验证**：重新推送 tag 触发 CI
   ```bash
   git tag -d vX.Y.Z && git push origin :refs/tags/vX.Y.Z
   git tag vX.Y.Z && git push origin vX.Y.Z
   ```

5. **检查 Release**：等待 CI 完成后，确认 Release 页面有 `.sig` 文件

## 方案 B：手动签名（无需 GitHub Secret）

如果 CI 已构建完成但缺少签名，可以手动签名。

### 前提条件

需要安装 Tauri CLI：

```bash
# 方式 1：通过 cargo（需 Rust 环境）
cargo install tauri-cli

# 方式 2：通过 npm（需 Node.js 环境）
npm install -g @tauri-apps/cli
```

> 注意：当前环境（Kimi Work）**没有** Tauri CLI，需要用户在本地安装。

### 步骤

1. **下载已发布的安装包**（从 GitHub Release 页面）

2. **签名**：
   ```bash
   tauri signer sign --private-key-path keys/plotline.key -- Plotline_X.Y.Z_x64-setup.exe
   ```
   这会生成 `Plotline_X.Y.Z_x64-setup.exe.sig` 文件。

3. **生成 latest.json**：
   ```bash
   tauri signer sign --private-key-path keys/plotline.key --output latest.json -- Plotline_X.Y.Z_x64-setup.exe
   ```
   或手动编辑 `releases/vX.Y.Z/latest.json`，将 `<PLACEHOLDER>` 替换为签名内容。

4. **上传到 GitHub Release**：
   - `.sig` 文件 → 作为 asset 上传
   - `latest.json` → 覆盖现有文件（确保文件名正确，以便 `releases/latest/download/latest.json` 生效）

## 方案 C：禁用签名（不推荐）

如果暂时不需要自动更新，可以移除签名验证：

```diff
// src-tauri/tauri.conf.json
"plugins": {
  "updater": {
    "active": true,
    "endpoints": [...],
    "dialog": false,
-   "pubkey": "..."
  }
}
```

> ⚠️ **风险**：客户端不会验证更新包的真实性，存在中间人攻击风险。仅用于开发测试。

## 常见问题

### Q: CI 构建成功但 Release 没有 `.sig` 文件
**A**: `TAURI_SIGNING_PRIVATE_KEY` GitHub Secret 未配置或已失效。按方案 A 重新配置。

### Q: 旧版本提示 "更新失败，签名验证不通过"
**A**: `latest.json` 中的签名与安装包不匹配。重新签名并上传。

### Q: 密钥丢失怎么办？
**A**: 
1. 生成新密钥对：`tauri signer generate`
2. 更新 `tauri.conf.json` 中的 `pubkey`
3. 将新私钥配置到 GitHub Secret
4. 旧版本客户端无法自动更新（pubkey 不匹配），需要手动重新安装
